# COLLAB.md — 클로드 코드 ↔ 코워크 협업 프로토콜

두 AI 에이전트가 **깃허브를 공유 지점**으로 두고 비동기 협업하기 위한 규칙. 목표는
"멈추지 않고" 진행하되 서로의 작업을 덮어쓰지 않는 것.

## 역할 분담 (소유 영역으로 충돌 방지)

| 에이전트 | 역할 | 주 소유 영역 |
|---|---|---|
| **클로드 코드** | 주력 코드 빌더 | `packages/**`, `apps/**`, 테스트 |
| **코워크** | 오케스트레이터: 기획·우선순위·리서치·리뷰·문서·조율·보고 | `docs/**`, `SPEC.md`, `BACKLOG.md`, `COLLAB.md`, `PROGRESS.md` |

- 코드는 클로드 코드가 씀. 코워크는 코드 영역을 직접 대폭 수정하지 않고, 리뷰 코멘트나
  작은 제안 PR로만 관여한다.
- 기획/문서/백로그는 코워크가 관리. 클로드 코드는 이 영역을 임의로 갈아엎지 않는다.

## 워크플로

1. **할 일 큐 = `BACKLOG.md`** (깃허브 Issues로 승격 가능). 각 항목엔 소유자 표기.
2. 작업 전 **`git pull`**로 최신 동기화. 항상 **브랜치**에서 작업(`feat/`, `fix/`, `docs/`).
3. 작업 후 **PR** 생성. 자기 자신이 즉시 머지하지 말고, 상대(또는 사람)의 리뷰를 거친다.
   - 코워크가 클로드 코드의 코드 PR을 리뷰(독립적 검토).
   - 사소한 문서 PR은 코워크가 스스로 머지 가능.
4. 머지 후 `PROGRESS.md`에 로그 한 줄(누가·무엇을·다음).

## 충돌이 나면

- 같은 파일을 동시에 건드려 충돌하면, **코드 = 클로드 코드 우선, 문서/기획 = 코워크 우선**으로
  해소. 애매하면 사람(경빈)에게 판단 요청.

## 환경 제약 (중요)

- 코워크 자율 세션은 **격리된 클라우드 샌드박스**에서 돈다. 이 샌드박스는 GitHub **API가
  차단**돼 있다(gh 설치·curl 모두 불가). **순수 git(fetch/pull/push)만** 가능하다.
- 따라서 **코워크는 PR을 열거나 병합할 수 없다.** 클로드 코드 Routine은 정식 인프라에서
  GitHub 도구를 내장해 돌기 때문에 PR 생성·병합이 가능하다.

## 병합(merge) 정책 (강한 브랜치 보호 기준)

- `main`은 브랜치 보호로 잠근다: **PR 필수 + CI(GitHub Actions) 초록 필수**. 직접 push 불가.
- **클로드 코드**: `claude/*` 브랜치에서 구현 → PR 생성. CI 초록이면 **클로드 코드 또는 사람이 병합.**
- **코워크**: main에 직접 못 쓴다. 산출물(리뷰 노트·리서치·문서·폴백 구현)을 `cowork/orchestrator`
  브랜치에 쌓아 push한다. 이 브랜치의 병합은 사람 또는 클로드 코드가 한다.
- 코워크의 PR 리뷰는 "읽고 판단"까지다: 브랜치를 git으로 받아 diff·build·test를 확인하고
  권고를 `docs/reviews/`에 남긴다. 실제 병합 클릭은 하지 않는다(못 한다).
- 사람(경빈)이 최종 병합 게이트를 쥐는 것이 기본이다("리뷰 후 배포" 원칙).

## 현재 상태 (2-에이전트 체제 가동)

- 원격 `origin/main` 연결. CI 워크플로 있음(.github/workflows/ci.yml). 클로드 코드 Routine =
  주력 빌더(매시), 코워크 트리거 v2 = 오케스트레이터(매시 :45, 브랜치 전용).
- **강건성(멈추지 않기)**: 클로드 코드가 한동안 정지한 듯하면, 코워크가 `cowork/orchestrator`
  브랜치에서 다음 항목을 직접 구현·테스트해 push한다(병합은 사람 몫). 진행이 멈추지 않게.
