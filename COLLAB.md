# COLLAB.md — 클로드 코드 ↔ 코워크 협업 프로토콜

두 AI 에이전트가 **깃허브를 공유 지점**으로 두고 비동기 협업하기 위한 규칙. 목표는
"멈추지 않고" 진행하되 서로의 작업을 덮어쓰지 않는 것.

## 역할 분담 (소유 영역으로 충돌 방지)

| 에이전트 | 역할 | 주 소유 영역 |
|---|---|---|
| **클로드 코드** | 주력 코드 빌더 | `packages/**`, `apps/**`, 테스트 |
| **코워크** | 오케스트레이터: 기획·우선순위·리서치·리뷰·문서·조율·보고 | `docs/**`, `SPEC.md`, `BACKLOG.md`, `COLLAB.md`, `PROGRESS.md` |

- 코드는 클로드 코드가 씀. 코워크는 코드 영역을 직접 대폭 수정하지 않고, 리뷰 코멘트나
  작은 제안 PR로만 관여한다.
- 기획/문서/백로그는 코워크가 관리. 클로드 코드는 이 영역을 임의로 갈아엎지 않는다.

## 워크플로

1. **할 일 큐 = `BACKLOG.md`** (깃허브 Issues로 승격 가능). 각 항목엔 소유자 표기.
2. 작업 전 **`git pull`**로 최신 동기화. 항상 **브랜치**에서 작업(`feat/`, `fix/`, `docs/`).
3. 작업 후 **PR** 생성. 자기 자신이 즉시 머지하지 말고, 상대(또는 사람)의 리뷰를 거친다.
   - 코워크가 클로드 코드의 코드 PR을 리뷰(독립적 검토).
   - 사소한 문서 PR은 코워크가 스스로 머지 가능.
4. 머지 후 `PROGRESS.md`에 로그 한 줄(누가·무엇을·다음).

## 충돌이 나면

- 같은 파일을 동시에 건드려 충돌하면, **코드 = 클로드 코드 우선, 문서/기획 = 코워크 우선**으로
  해소. 애매하면 사람(경빈)에게 판단 요청.

## 병합(merge) 정책

- 클로드 코드는 `claude/*` 브랜치에서 작업하고 `main`으로 PR을 연다.
- **코워크(오케스트레이터)가 그 PR을 검토·병합**한다. 절차: 브랜치를 fetch → 코드 검토
  (정확성/품질/범위) → `pnpm install && pnpm build && pnpm test` 통과 확인 →
  통과하고 문제없으면 `main`에 병합(fast-forward 또는 --no-ff) 후 push(=PR 자동 종료).
  검토 요약은 `docs/reviews/`에 남긴다.
- 테스트 실패거나 위험/되돌리기 어려운 변경이면 **병합하지 말고** `PROGRESS.md`와
  `docs/reviews/`에 사유를 적어 사람/클로드 코드가 고치도록 남긴다.
- 코워크 소유 문서(`docs/**`, `README.md`, `ARCHITECTURE.md`, `ROADMAP.md`, `BACKLOG.md`
  등)는 코워크가 `main`에 직접 커밋해도 된다(코드와 충돌 영역이 다름).

## 현재 상태 (2-에이전트 체제 가동)

- 깃허브 원격 연결됨(`origin/main`). 클로드 코드 Routine = 주력 빌더(매시), 코워크 트리거 =
  오케스트레이터(매시, 분offset 다르게 스태거).
- **강건성(멈추지 않기)**: 만약 클로드 코드가 한동안 아무 브랜치/커밋도 안 내면(정지 추정),
  코워크가 "빌더 폴백"으로 직접 `cowork/*` 브랜치에서 다음 항목을 구현·테스트·병합해
  진행이 멈추지 않게 한다. 클로드 코드가 정상 작동 중이면 코워크는 리뷰·문서·리서치에 집중.
